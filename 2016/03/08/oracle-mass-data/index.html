Oracle海量数据笔记1

2016-03-08
database

背景介绍及理论基础

关系型数据库：Oracle，MySQL，SQL Server
NoSQL数据库：
键值对（内存）Memcached
键值对（磁盘）Redis
列式 Cassandra HBase Google BigTable
文档 MongoDB（面向集合）
图 Neo4j

Hadoop：分布式处理软件框架

选择：事务性，数据一致性，性能，水平扩展性，并行计算

ACID（RDBMS理论基石）：
原子性(Atomicity)，一致性(Consistency)，隔离性(Isolation)，持久性(Durability)

CAP理论：
数据一致性(Consistency)，高可用性(Availability)，分区容错性(Partition Tolerance)

NWR法则：
N: 复制的节点数量
R: 成功读操作的最小节点数
W: 成功写操作的最小节点数
当:
R+W>N 可以保证强一致性
R+W<=N 无法保证数据的一致性
读写平衡:
W=Q R=Q Q=N/2+1 比如：N=3 R=2，W=2

分布式数据库架构设计：
主机数量，数据冗余，系统监控，水平扩展，备份及恢复，容灾

影响分布式数据库数量的因素：
系统容量，查询性能，加载性能，恢复时间

分布式系统中间件架构设计：
统一API，负载均衡，数据查询，数据加载，服务监控，容错和扩展

常用技术

分区

功能：增强可用性，维护方便，均衡I/O，改善查询性能。

类型：
范围分区 range partition
哈希分区 hash partition（无法删除，只能合并）
列表分区 list partition
组合分区

索引

B树索引
适用：OLTP；主键或者唯一性约束；键值重复率低的字段
缺点：不适合键值重复率高的字段

位图索引
适用：OLAP；重复率高的数据；高重复率键值上的COUNT，OR，AND 等操作
优点：节省空间
缺点：不适合键值重复率低的字段和DML操作

全文索引
适用：模糊查询
缺点：占用空间；有bug；难维护

并行

使用：
对象的属性：表；索引；
SQL显式执行：hint

并行查询的效率：服务器配置越高，资源越充裕，越适合使用并行

考虑使用并行：多CPU；硬件I/O充足；系统负载轻；

RAC上的并行：
指定执行并行操作的实例；不允许跨实例执行并行操作；

并行DML
直接加载：sql*loader；/*+ append */
insert，update，merge：alter session enable parallel dml

压缩

使用：表；分区；索引

作用：节省空间；提高数据处理效率；减少内存和I/O开销；

原理：替换重复值

数据压缩不适用于OLTP
数据块内的行数变多引起事务的争用
处理数据需要更长的时间：数据插入，修改的时间变长；索引键值更新的时间变长；

数据压缩的应用场景：
静态数据的处理；
海量数据的处理：减少处理的数据块，减少I/O和内存的消耗；
历史数据进行数据压缩存储：减少磁盘空间，提高查询效率；
数据备份前进行压缩处理：减少数据的备份和恢复时间；
